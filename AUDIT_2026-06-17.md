## CRITICAL — found & FIXED

### 1. Truncated `change_project_dir` endpoint (`backend/api/settings.py`)

The file ended mid-statement at `settings.project_dir = str` (no closing call, no commit, no return). It parsed clean only because `str` is a valid builtin reference, so neither `ast.parse` nor `tsc` would ever catch it — this is exactly the Edit-tool Windows-mount truncation pattern noted in the project handover.

**Impact (before fix):** Calling "Change project directory" with *move data* enabled would physically `shutil.move` all project data to the new folder, then **fail to persist the new path and never commit** — leaving the DB pointing at the now-empty old directory. Data-loss / orphaned-data risk. The endpoint also returned no body despite declaring `response_model=ChangeProjectDirResponse`.

Notably, this truncation was present in git HEAD as well (working tree was clean), so it had shipped.

**Fix applied** (completes the statement, persists, returns the declared model — matching the commit pattern used elsewhere in the file at lines 614–615, 1553–1554):

```python
    settings.project_dir = str(new_path)
    session.add(settings)
    await session.commit()
    await session.refresh(settings)
    logger.info(f"Project directory changed: {old_path} -> {new_path} ({result_msg})")
    return ChangeProjectDirResponse(
        success=True, message=result_msg,
        old_path=str(old_path), new_path=str(new_path),
    )
```

Verified with `ast.parse` clean. The restart-to-apply behavior is by design and already surfaced in the UI (`SettingsPage.tsx` tells the user to restart), so no further change needed there.

## MINOR — safe improvements (non-breaking, optional)

### 2. Deprecated `asyncio.get_event_loop()` — `settings.py:1749`, `settings.py:1829`
Works today, but `get_event_loop()` with no running loop is deprecated (Python 3.12+) and will emit `DeprecationWarning`. Both call sites are `loop.run_in_executor(None, fn)`, so the cleanest swap is `await asyncio.to_thread(fn)` (drops the loop handle entirely). Pure cleanup, no behavior change.

### 3. Stray `console.log` / `console.debug` in frontend (13 occurrences)
Left in `frontend/src`. Harmless but noisy in the browser console; consider stripping or gating behind a debug flag for a cleaner production console.

### 4. Stale duplicate build directory — `frontend/dist_new/` (2.6 MB)
Sits alongside the real `frontend/dist/` (3.8 MB). Looks like a leftover from a build experiment. Safe to delete if not referenced by the run scripts (the empty top-level `New folder/` is likewise removable). Housekeeping only.

### 5. `settings.py` style consistency
After the fix the file is fully valid, but it's the one file that previously shipped truncated — worth keeping an eye on given the Edit-tool corruption history. When editing it, prefer the heredoc/Python-rewrite path rather than the Edit tool.
